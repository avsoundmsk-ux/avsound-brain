# STATE_AND_RECOVERY.md

State Manager: устойчивость к обрывам. Повторный запуск продолжает с последней необработанной строки, не начиная Excel заново. (В этой работе обрывы сессии — главная боль, поэтому модуль критичен.)

# Что хранить (состояние запуска)
- `run_id` (метка времени запуска);
- путь и **хэш Excel-файла** (md5) — чтобы понять, тот же файл или новый;
- режим (dry-run / live);
- по каждой строке:
  - `row_key` (см. ниже);
  - `status`: `pending | done | failed | skipped | manual_review`;
  - `product_id` (если найден/создан);
  - `action`: matched / created / updated / skipped;
  - `error` (если failed);
  - `timestamp`.

# Ключ строки (row_key)
Артикула/SKU в Excel нет → ключ = **хэш(норм. название + бренд + категория)**. Дополнительно хранить номер строки Excel (для отчёта). Один и тот же товар в разных файлах даёт одинаковый row_key → не дублируется.

# Где хранить
- `output/<run_id>/state.json` — состояние конкретного запуска;
- `state/progress.json` — глобальный прогресс по row_key (история обработки между запусками, источник для «resume» и идемпотентности).

# Идемпотентность
- Повторный запуск того же Excel НЕ создаёт дубликаты.
- Строки со статусом `done` пропускаются (если не задан `--force`).
- `failed` — повторять при следующем запуске.
- `manual_review` — НЕ трогать автоматически (ждёт человека).
- `skipped` (пустая строка/секция) — пропускать.

# Флаги
- `--resume` (по умолчанию ВКЛ): продолжить с pending/failed, пропустить done.
- `--force`: переобработать всё, включая done (с осторожностью; всё равно проходит проверки безопасности и аудит).
- `--only <row_keys|ids>`: точечная переобработка.
- `--limit N`: ограничить число товаров за запуск (пилот).

# Чекпоинт
- Состояние писать на диск **после каждого товара** (не в конце), чтобы обрыв терял максимум один товар.
- Запись атомарная (temp-файл → rename).

# Поведение при обрыве
- Незавершённый товар остаётся `pending` → переобработается.
- Если товар успел измениться на сайте, но state не записался — повторная обработка безопасна: матчинг найдёт его же (по row_key/названию), обновит идемпотентно, аудит зафиксирует.

# Открытые параметры
- хранить ли глобальный progress.json в git (история) или только локально;
- TTL/ротация старых state-файлов.
