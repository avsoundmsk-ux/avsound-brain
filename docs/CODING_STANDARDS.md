# CODING_STANDARDS.md

# Стандарты разработки

Данный документ определяет обязательные правила написания кода.

Все новые модули должны соответствовать данным требованиям.

---

# Главная цель

Писать код так, чтобы через несколько лет его можно было легко поддерживать, расширять и читать.

Код должен быть понятен другому разработчику без дополнительных объяснений.

---

# Основные принципы

Каждый файл выполняет только одну задачу.

Каждая функция выполняет только одну задачу.

Каждый класс отвечает только за одну область.

Избегать больших файлов.

Избегать больших функций.

---

# Размер файлов

Желательно:

до 300 строк.

Максимум:

500 строк.

Если файл становится больше —

разделить его на несколько модулей.

---

# Размер функций

Желательно:

до 30 строк.

Максимум:

50 строк.

Если функция становится длиннее —

разделить ее.

---

# Имена

Использовать понятные имена.

Хорошо:

find_product()

generate_description()

upload_images()

Плохо:

run()

do_it()

work()

temp()

test()

---

# Комментарии

Комментарии писать только там, где действительно сложная логика.

Не комментировать очевидные вещи.

Код должен объяснять себя сам.

---

# Повторение кода

Не копировать одинаковый код.

Если логика повторяется —

создать отдельную функцию.

---

# Константы

Все константы хранить отдельно.

Не использовать "магические числа".

Плохо:

timeout = 17

Хорошо:

DEFAULT_TIMEOUT = 17

---

# Конфигурация

Все настройки хранить в config.

Все API-ключи хранить только в .env.

Никогда не записывать ключи в исходный код.

---

# Работа с API

Для каждого API создать отдельный клиент.

Например:

WooCommerceClient

KieClient

GoogleImageClient

OpenRouterClient

Никогда не смешивать разные API в одном модуле.

---

# Обработка ошибок

Любая ошибка должна:

* логироваться;
* не завершать программу полностью;
* попадать в отчет.

После ошибки обработать следующий товар.

---

# Логирование

Каждый важный этап писать в лог.

Например:

Старт обработки товара.

Поиск характеристик.

Поиск изображений.

Генерация KIE.ai.

Загрузка WooCommerce.

Удаление старых фото.

Завершение.

---

# Dry Run

Любая новая функция сначала должна поддерживать Dry Run.

Только после проверки разрешается использовать Live Run.

---

# Тестирование

Перед объединением нового кода проверить:

✓ код запускается;

✓ нет синтаксических ошибок;

✓ нет неиспользуемых импортов;

✓ нет дублирования функций;

✓ нет циклических импортов.

---

# Производительность

Не выполнять повторные запросы без необходимости.

Использовать кеширование там, где это возможно.

Не загружать одинаковые изображения повторно.

---

# Git

Каждая законченная функция должна сопровождаться отдельным коммитом.

Сообщения коммитов должны быть понятными.

Например:

Add image validator

Improve WooCommerce uploader

Fix Excel parser

Плохо:

fix

update

test

---

# Документация

После добавления новой функции обновлять соответствующую документацию в папке docs.

Документация должна всегда соответствовать реальному коду.

---

# Архитектура

Не нарушать структуру проекта.

Не создавать случайные папки.

Не складывать разные задачи в один файл.

---

# AI

Перед написанием нового кода сначала искать возможность переиспользовать существующие модули.

Не переписывать работающий код без необходимости.

Если существует более простое решение — использовать его.

---

# Главный принцип

Код должен быть:

* безопасным;
* читаемым;
* расширяемым;
* модульным;
* предсказуемым.

Любое решение должно упрощать проект, а не усложнять его.
