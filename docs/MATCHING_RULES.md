# MATCHING_RULES.md

Правила поиска товара на сайте и определения «уверенного» совпадения.

ВАЖНО (факт по `Копия склад (8).xlsx`): в Excel НЕТ артикула и НЕТ SKU. Единственный надёжный текст — колонка E «название» (бренд+модель внутри) + колонка D «категория». Поэтому матчинг строится на **бренд + модель + нормализованное название + категория**, с защитой от дублей.

---

# 1. Извлечение признаков из строки Excel

Из строки получаем:
- **brand** — из секции-заголовка блока (Quart/JBL/Nakamichi/Ural) либо из начала названия;
- **name_raw** — колонка E как есть;
- **model** — name_raw минус префиксы-типы (Усилитель/Сабвуфер/Компонентная акустика/Твитеры/Магнитола/…) и минус бренд;
- **category** — колонка D;
- **доп. признаки** из модели: размер (10"/12"/16см/6x9), импеданс (D2/D4/2Ω/4Ω), версия (ver.2/v.2/mkii), мощность.

# 2. Нормализация названия (norm)

Применять и к Excel, и к названиям на сайте одинаково:
1. lower-case;
2. убрать бренд из строки модели (сравнение по модели отдельно);
3. убрать слова-типы (усилитель, акустика, сабвуфер, динамик, твитер, магнитола, компонентная, коаксиальная …);
4. унифицировать версии: `ver.2 = v.2 = вер.2 = v2`; `mkii = mk2 = мк2`;
5. унифицировать единицы: `6,5 = 6.5`; `16см = 165`; `6x9 = 69`;
6. убрать пунктуацию, лишние пробелы;
7. латиница/кириллица: привести визуально-одинаковые модели (опц., с осторожностью).

Ключевые токены (размер/импеданс/версия) НЕ выбрасывать — они различают варианты (D2≠D4, 10"≠12").

# 3. Порядок поиска кандидатов на сайте

1. точное совпадение `norm(model)` в пределах того же бренда;
2. точное совпадение `norm(name)`;
3. fuzzy по `norm(name)` (token_sort_ratio);
4. fuzzy в пределах бренда + категории.

# 4. Скоринг совпадения (0–100)

Складывать баллы:
- бренд совпал: +35 (обязательное условие — без совпадения бренда матч не засчитывается);
- модель `norm` точно совпала: +45;
- fuzzy модели ≥ 92: +35; 85–91: +20; <85: 0;
- категория совпала/совместима: +10;
- все «различающие» токены совпали (размер, импеданс, версия): +10;
- конфликт различающего токена (D2 vs D4, 10" vs 12", ver.2 vs ver.1): −40 (штраф).

# 5. Решение

- **score ≥ 90 и единственный такой кандидат** → УВЕРЕННОЕ совпадение → обновлять.
- **score 75–89**, или несколько кандидатов с близкими баллами (разница < 10) → СОМНИТЕЛЬНО → НЕ менять, в отчёт «Требуется ручная проверка».
- **score < 75** → товар на сайте не найден → переход к проверке на дубликаты перед созданием.

Обязательное правило: **бренд должен совпасть всегда**. Совпадение только по названию без совпадения бренда → не матч.

# 6. Защита от дублей ПЕРЕД созданием

Перед созданием нового товара выполнить повторный поиск по ВСЕМ признакам:
- по norm(model) во всём каталоге (не только в бренде);
- по norm(name);
- по «бренд + модель»;
- по различающим токенам.

Если найден кандидат со score ≥ 75 → НЕ создавать, отправить в ручную проверку (возможный дубль).
Создавать товар только если уверенно НИЧЕГО похожего нет.

# 7. Несколько вариантов одной модели

Если модель существует в вариантах (импеданс D2/D4, размер 10"/12") — это РАЗНЫЕ товары. Совпадение засчитывать только при совпадении различающего токена. При отсутствии токена в Excel или на сайте → ручная проверка (не угадывать вариант).

# 8. Что логировать по каждому матчу (в отчёт/аудит)
- строка Excel, извлечённые brand/model/category;
- топ-3 кандидата с их score;
- принятое решение (обновить / создать / ручная проверка) и причина.

# 9. Открытые параметры (согласовать)
- библиотека fuzzy и пороги (предложено token_sort_ratio, 92/85);
- канон бренда (MB Quart vs Quart);
- список слов-типов для очистки (вести в коде/конфиге, не хардкодить).
