# AUDIT_RULES.md

Система полного аудита всех действий AI. Модуль: `audit_manager`. Один из важнейших документов проекта.

# Цель
В любой момент ответить: ЧТО изменилось, КОГДА, ПОЧЕМУ, КТО, из какого Excel, какие данные были ДО и какие стали ПОСЛЕ. Журнал — будущая основа Rollback.

# Главный принцип
Ни одно изменение не происходит без записи в аудит. Любое действие AI оставляет полную историю. По журналу можно восстановить цепочку изменений любого товара с первого дня.

---

# Шапка записи (для каждого изменения)
- дата и время (ISO 8601, таймзона);
- run_id;
- версия программы;
- версия документации;
- пользователь = `AI Agent` (+ модель);
- режим (Dry Run / Live Run);
- имя Excel-файла + хэш;
- номер строки Excel;
- row_key (STATE_AND_RECOVERY);
- Product ID;
- SKU;
- название товара;
- бренд;
- категория;
- ссылка на товар;
- действие: `created | updated | skipped | manual_review | deleted_photo | error`;
- причина (напр. «Импорт Excel»).

# Изменения полей (diff «было/стало»)
Для каждого изменённого поля: имя поля, старое значение, новое значение, причина.
Отслеживать: цена, описание, краткое описание, характеристики, категория, бренд, SEO (title/desc/focus), название, остаток, статус наличия, фотографии.
```
Цена   было: 18990   стало: 17990   причина: Импорт Excel
```

# Фотографии (по каждой операции)
- старые URL; новые URL;
- attachment_id старых; attachment_id новых;
- источник найденного фото (страница/магазин);
- источник после генерации (KIE / dark_bg);
- число попыток KIE.ai;
- стоимость генерации (cr/$);
- итоговый статус (ok / manual_review / failed).
Перед удалением старых фото их URL+attachment_id сохраняются ДО удаления (Backup Manager); без бэкапа удаление запрещено.

# Создание товара
- дата создания; исходная строка Excel + row_key; Product ID; полный снимок всех заполненных полей.

# Обновление товара
- полный список изменённых полей с diff.

# Ошибки
- время; этап; модуль; описание; рекомендуемое действие; Product ID (если есть).

# Запуски
- каждый запуск → `run_id` (`YYYY-MM-DD_HH-MM-SS`);
- все записи привязаны к run_id;
- summary: счётчики created/updated/skipped/manual/errors, бюджет, время.

---

# Структура хранения
```
/audit/
  2026-06-27_10-42-51/        ← run_id
    audit.json     ← все записи запуска (полные, pre/post снимки)
    audit.xlsx     ← человекочитаемо
    audit.csv      ← плоский экспорт
    changes.json   ← только diff полей
    errors.json    ← ошибки
    images.json    ← операции с фото
    summary.json   ← сводка запуска
```
- хранить **бессрочно**;
- история по товару = фильтр по Product ID / row_key через все run-папки.

# Порядок записи (обязательный)
1. снять **pre-state** (полный снимок товара) → аудит;
2. выполнить изменение (только Live Run);
3. снять **post-state** → аудит + diff;
4. при удалении фото — backup URL/attachment_id ДО удаления.
В Dry Run: записывать ПЛАНИРУЕМЫЙ diff (было/станет) без применения.

# Rollback (будущее, сейчас НЕ реализуем)
Структура спроектирована под восстановление любого предыдущего состояния по pre-state снимкам. Сейчас — только сбор истории.
