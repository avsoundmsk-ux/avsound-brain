## Agent Orchestration — оркестратор агентов

Цель: под каждую задачу автоматически подобрать специалиста и НЕ перегружать контекст. Агенты дороже плагинов — зовём только по делу.

### Шаг 1. Определи категорию задачи (ДО любого действия)

Сначала классифицируй задачу в одну или несколько категорий, и только потом выбирай агента:

`business` · `marketing` · `seo` · `frontend` · `backend` · `python` · `php` · `react` · `vite` · `woocommerce` · `telegram` · `ai` · `automation` · `testing` · `performance` · `security` · `build` · `documentation`

### Шаг 1.5. Confidence + неопределённость (ОБЯЗАТЕЛЬНО перед выбором агента)

Перед выбором агента вычисли confidence (0–100%) — уверенность в категории и нужности агента:

| Confidence | Действие |
|---|---|
| **90–100% (высокая)** | запускать агента автоматически |
| **70–89% (средняя)** | можно запускать; но при риске ошибки — сначала минимальные уточняющие вопросы |
| **ниже 70% (низкая)** | НЕ запускать; не угадывать — задать минимальные уточняющие вопросы, потом решить |

**Правила неопределённости (жёсткие):**
1. Уверенность **не высокая** → НЕ угадывай агента/категорию. Задай **минимум** уточняющих вопросов (1–2), затем выбирай.
2. Категорию задачи **НИКОГДА** не угадывать.
3. Задача **бизнесовая / markdown / консультация / база знаний** → код-агентов НЕ запускать вообще, независимо от confidence.
4. **Нет подходящего агента** под задачу → предложи создать нового (роль + источник), но **НЕ создавай без подтверждения** пользователя.

### Шаг 3. Правила запуска (жёсткие)

1. **Нет агента в строке → не зови никого.** Работай сам. Не выдумывай агента.
2. **Бизнес-задачи (business/marketing/seo/знания/документация) НИКОГДА не запускают код-агентов** (build-error-resolver, python-reviewer, php-reviewer).
3. **Код-агент только при явной код-причине**: есть `.py`/`.php`/`.jsx`, ошибка сборки, прогон скрипта. Нет кода — нет код-агента. Слова «PROACTIVELY / MUST BE USED» в описаниях агентов игнорируются в пользу этого правила.
4. **Несколько подходящих агентов → запускай ПО ОЧЕРЕДИ (последовательно), не параллельно.** Контекст и стоимость под контролем.
5. **Дубликат функции → выбери ОДНОГО, наиболее подходящего:**
   - Python: правка `tools/*.py` перед массовым прогоном → `tester` (бережёт деньги KIE/каталог). Ревью качества кода → `python-reviewer`. Не оба сразу.
   - security → local `security-reviewer` (не сторонние).
   - performance → local `performance-optimizer`.

### Шаг 4. Приоритет источника (при выборе из нескольких)

1. **Local** — агенты проекта в `.claude/agents/` (главный приоритет).
2. **Global / проверенные plugin-агенты** — только если local нет.
3. **Сторонние** (ECC и пр.) — в последнюю очередь, и только после проверки + переноса в local.

### Шаг 5. Расширяемость

Новый агент в будущем → **добавь одну строку в таблицу Шага 2**. Правила Шагов 3–4 переписывать не нужно — они работают для любого агента.

**НИКОГДА не говори "зайди на сайт", "открой браузер", "проверь вручную" — если есть инструмент который это умеет.**

Порядок действий при любой задаче:
1. **Сначала попробуй сам** — используй Playwright, Firecrawl, Claude in Chrome, computer-use, curl, bash
2. Если первый инструмент не сработал — **пробуй следующий**, не сдавайся
3. Только если все инструменты исчерпаны и есть техническое ограничение — скажи об этом и объясни почему

Таблица: что делать вместо "зайди сам":

| Вместо | Делай |
|---|---|
| "зайди на сайт X" | Firecrawl scrape / Playwright / Claude in Chrome |
| "открой страницу" | `firecrawl_scrape` или `playwright_navigate` |
| "проверь контент сайта" | `firecrawl_scrape` + анализ |
| "посмотри что там написано" | `firecrawl_scrape` или `WebFetch` |
| "сделай скриншот" | `playwright_screenshot` или computer-use |
| "заполни форму" | Playwright / Claude in Chrome |
| "проверь в поиске" | `firecrawl_search` или Brave Search |

**Правило максимального результата**: если задача не получилась — разложи на шаги, попробуй каждый шаг по-другому. Не останавливайся на первой неудаче. Пользователь ждёт результат, а не объяснение почему не вышло.
