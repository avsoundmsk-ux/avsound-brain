## 1. Авторизация

### Применимый закон
**ФЗ № 406-ФЗ от 31.07.2023** (поправки в 149-ФЗ «Об информации»). С **1 января 2025** авторизация пользователей на сайтах российских владельцев информационных ресурсов допускается только через:
- номер телефона российского оператора связи;
- ЕСИА (Госуслуги);
- ЕБС (Единая биометрическая система);
- иную информсистему, владелец которой — гражданин РФ или российское юрлицо.

**Штрафы** (КоАП): граждане 10–20 тыс., должностные 30–50 тыс., **юрлица 500–700 тыс. руб.**

### Разбор способов

| Способ | Статус для РФ-сервиса | Комментарий |
|---|---|---|
| Вход по **телефону РФ** | ✅ Разрешён, **рекомендуется как основной** | Соответствует 406-ФЗ напрямую |
| **Госуслуги (ЕСИА)** | ✅ Разрешён, рекомендуется | Прямо назван в законе; повышает доверие |
| **Яндекс ID** | ✅ Допустим | Владелец — росс. юрлицо (иная инфосистема по 406-ФЗ) |
| **VK ID** | ✅ Допустим | Владелец — росс. юрлицо |
| Вход по **e-mail** (свой пароль) | ⚠️ Серый | Сам по себе email-логин не «иностранный сервис»; рискует трактовка через подтверждение письмом на зарубежном ящике. Безопасно — только как доп. способ при наличии РФ-способа |
| Регистрация **с Gmail-адреса** | ⚠️ Риск | Запрет касается авторизации *через иностранный сервис*; ввод gmail-адреса в своей форме спорен, но РКН трактует широко. Не делать единственным способом |
| **Google OAuth** | ⛔ Запрещён | Иностранный сервис идентификации |
| **Apple ID** | ⛔ Запрещён | Иностранный сервис |
| **Facebook** (Meta) | ⛔ Запрещён + Meta признана экстремистской в РФ | Использовать нельзя |
| **Telegram-логин** | ⚠️ Риск | Владелец иностранный; как способ авторизации по 406-ФЗ небезопасен. Допустимо как канал уведомлений, не как идентификация |
| **Microsoft / GitHub / Discord OAuth** | ⛔ Запрещены | Иностранные сервисы |

### Что делать
В коде соц-OAuth нет — **ничего сносить не нужно**. Нужно **добавить** РФ-способ: вход по телефону РФ (SMS-код) и/или ЕСИА. Email+пароль можно оставить как дополнительный, но не как единственный механизм для публичного запуска.

**План миграции авторизации (на этап реализации, не сейчас):**
1. Подключить SMS-провайдера РФ (например, через ЮKassa/российские SMS-агрегаторы) → вход по телефону.
2. (Опционально, повышает доверие) интеграция ЕСИА/Госуслуги.
3. Email+пароль — оставить вспомогательным; подтверждение email вести через **российский** SMTP (см. §8).
4. Не подключать Google/Apple/Facebook/Microsoft/GitHub/Discord OAuth.

**Риск:** 🔴 Критический (прямой запрет 406-ФЗ, штраф до 700 тыс.). **Обязательно до запуска.**

## 2. Регистрация пользователей

- **Иностранные почтовые сервисы:** закон не запрещает пользователю *иметь* Gmail; запрещает строить *авторизацию* на иностранном сервисе (§1). Свою регистрацию можно разрешать с любым email, **но** основной способ входа должен быть РФ-совместимым.
- **Согласие при регистрации:** ст. 9 152-ФЗ требует согласия субъекта на обработку ПДн. В текущей форме регистрации согласия нет — нарушение (см. §3, §4).
- **Штрафы:** обработка без согласия — ст. 13.11 КоАП; с **30.05.2025** размеры существенно повышены (см. §3).

**Риск:** 🔴 Критический. **Обязательно до запуска:** добавить чекбокс согласия + ссылки на документы, обеспечить РФ-способ входа.

## 3. Персональные данные

### Какие ПДн собирает проект (из схемы БД)
- **Идентификация/контакт:** email, имя (`user`), `image`.
- **Технические ПДн:** IP-адрес и User-Agent (`session`), IP (`admin_audit`). По практике РКН IP-адрес — персональные данные.
- **Платёжные:** суммы, валюта, внешний id платежа (`topups`); сами карты — на стороне платёжного провайдера.
- **Пользовательский контент:** промты и загруженные изображения (`jobs.prompt`, `jobs.params`). ⚠️ Если пользователь грузит фото людей для image/video-генерации — это может быть **биометрия** (ст. 11 152-ФЗ), требующая отдельного усиленного согласия.

### Где хранятся (факт)
Neon (Postgres) — за рубежом; S3 (R2/Backblaze) — за рубежом; промты/картинки уходят в ModelArk/BytePlus (Сингапур) и KIE.

### Ключевые требования
1. **Локализация — ст. 18 ч. 5 152-ФЗ.** Запись, систематизация, накопление, хранение, уточнение, извлечение ПДн граждан РФ должны вестись с использованием БД, **находящихся на территории РФ**. Хранение первичной БД на Neon/за рубежом — **прямое нарушение**.
2. **Уведомление РКН** о намерении обрабатывать ПДн (ст. 22 152-ФЗ) — с 01.09.2022 обязательно практически для всех операторов. С 30.05.2025 неуведомление — штраф юрлицу **100–300 тыс. руб.**
3. **Трансграничная передача — ст. 12 152-ФЗ.** Передача промтов/изображений в ModelArk (Сингапур), KIE, а также хранение на зарубежных Vercel/Hetzner/S3 — трансграничная передача. С 01.03.2023 требуется **уведомить РКН до начала передачи**; для стран без адекватной защиты — доп. условия.
4. **Согласие — ст. 9 152-ФЗ.** Сейчас отсутствует.

### Штрафы (с 30.05.2025, 420-ФЗ → КоАП)
- Утечка ПДн: оборотные штрафы **1–3% выручки**, мин. 25 млн, макс. 500 млн руб. (при повторности).
- Неуведомление РКН об утечке (в течение 24 ч): **1–3 млн руб.**
- Неуведомление о намерении обработки: **100–300 тыс. руб.**

**Риск:** 🔴 Критический (локализация + трансграничка + отсутствие согласия и уведомления). **Обязательно до запуска.**

## 6. Серверная инфраструктура

**Факт:** Vercel (США), Hetzner (Германия), Neon (США/ЕС), Cloudflare R2 / Backblaze (зарубеж).

### Требования
- **Локализация ПДн (ст. 18 ч. 5 152-ФЗ):** первичная БД с ПДн граждан РФ — на серверах в РФ. Neon/Hetzner/Vercel этого не обеспечивают.
- **Реестр хостинг-провайдеров РКН (406-ФЗ, с 01.02.2024):** размещать ресурс следует у провайдера из реестра РКН; иностранные хостинги (Vercel, Hetzner) в реестре отсутствуют.

### Рекомендация
Перенести **минимум** первичную БД ПДн и хранилище пользовательского контента в РФ. Подходящие провайдеры из реестра РКН: **Yandex Cloud, VK Cloud, SberCloud (Cloud.ru), Selectel, Timeweb, REG.RU**. Архитектурно: фронт+БД+S3+worker → российское облако. Зарубежные AI-провайдеры (ModelArk/KIE) допустимы только при оформленной трансграничной передаче и согласии (§3), либо замене на РФ-аналоги.

**Риск:** 🔴 Критический (локализация + реестр хостинга). **Обязательно до запуска** (как минимум БД ПДн и контент-хранилище).

## 8. Электронная почта

**Факт:** в примере — Resend/SendGrid (зарубежные SMTP), отправка пока заглушка ([email.ts](../src/auth/email.ts)).

- Транзакционные письма (verify/reset) сами по себе не запрещены, но через зарубежный SMTP письмо с данными пользователя = трансграничная передача ПДн (email — ПДн) → требует оформления (§3).
- Подтверждение email как **механизм авторизации** на иностранном ящике — зона риска 406-ФЗ (§1).
- Рекламные рассылки — только с согласием (ст. 18 ФЗ «О рекламе»), §4.

### Рекомендация
Использовать **российский** почтовый/SMTP-сервис: **Unisender, SendSay, Mailopost (RuSender), Yandex Postbox / Mail for Domain**. Подтверждение регистрации перенести на РФ-способ (SMS/телефон) как основной.

**Риск:** 🟠 Высокий (трансграничка + 406-ФЗ при email-верификации как входе). **До запуска** перейти на РФ-SMTP.

## 9. Cookies

**Факт:** используется сессионная cookie better-auth (строго функциональная). Баннера, формы согласия и журнала согласий нет.

- Функционально необходимые cookie (сессия авторизации) — по преобладающей трактовке не требуют отдельного согласия, но **уведомить** пользователя нужно.
- Если добавятся аналитика/маркетинг (Яндекс.Метрика и т.п.) — потребуется **активное согласие** и возможность отказа.

### Нужно
1. Cookie-баннер с уведомлением и ссылкой на Cookie Policy — ✅ да.
2. Согласие на cookies — ✅ обязательно, если появятся не-функциональные cookie (аналитика/реклама).
3. Журнал согласий — ⚠️ желательно (доказательство согласия для РКН).

**Риск:** 🟡 Средний (пока только функциональная cookie). Поднимется до 🟠 при добавлении аналитики.

## 10. Логи

**Факт:** `session` — IP + User-Agent; `admin_audit` — IP; `job_logs` — jsonb без срока хранения (TTL).

- Логировать IP/User-Agent для безопасности **законно** (легитимная цель), но это ПДн → должны попадать под Политику и храниться в РФ (§3, §6).
- **Срок хранения:** ст. 5 ч. 7 152-ФЗ — хранить не дольше, чем нужно для цели. У `job_logs` нет TTL (отмечено и в [SECURITY.md](../SECURITY.md) L-1) → задать политику ретенции и очистку.
- Нарушений «избыточного» логирования не видно (пароли/токены в логи не пишутся — это уже исправлено, см. SECURITY.md C-4).

**Риск:** 🟡 Средний. Нужны: запись о логировании IP в Политику + TTL/ретеншн для логов.

## 12. Дорожная карта исправлений

### 🔴 Критические — СТОП-факторы запуска (без них публиковать нельзя)
1. **R5 Уведомление РКН** об обработке ПДн (подать до начала обработки реальных данных).
2. **R4/R6 Локализация + трансграничка:** перенести первичную БД ПДн и контент-хранилище в РФ-облако (Yandex Cloud / VK Cloud / Selectel и т.п.); оформить уведомление о трансграничной передаче для зарубежных AI-провайдеров (или заменить на РФ).
3. **R3 Юр. документы:** Политика конфиденциальности, Пользовательское соглашение, Оферта, Cookie Policy — составить и опубликовать.
4. **R2 Согласие на ПДн:** отдельный непредотмеченный чекбокс + ссылки в форме регистрации; сохранять факт согласия.
5. **R1 РФ-авторизация:** добавить вход по телефону РФ (SMS) и/или ЕСИА; email+пароль — только вспомогательный.

### 🟠 Высокий приоритет — до запуска
6. **R7 Хостинг:** мигрировать фронт/worker к провайдеру из реестра РКН (вместе с п.2).
7. **R8 Платежи:** не активировать Stripe для РФ; основной способ — ЮKassa (+ Т-Банк/CloudPayments опц.); подключить чеки 54-ФЗ.
8. **R6 Трансграничка:** уведомление РКН до начала передачи данных за рубеж (AI, любые зарубежные подрядчики).
9. **R12 Биометрия:** если в сценарии есть загрузка фото людей — отдельное согласие на биометрию **или** запрет такого контента в правилах.

### 🟡 Средний приоритет — желательно к запуску, допустимо сразу после
10. **R9 Email:** перейти на российский SMTP (Unisender/SendSay/RuSender); верификацию увести на SMS.
11. **R10 Cookies:** добавить баннер-уведомление + Cookie Policy; журнал согласий при появлении аналитики.
12. **R13 Рассылки:** отдельное согласие на рекламные рассылки (если планируются).

### 🟢 Низкий приоритет — после запуска
13. **R11 Логи:** TTL/ретеншн для `job_logs` и сессий (уже в бэклоге SECURITY.md L-1, P7).
14. Назначить ответственного за обработку ПДн, утвердить внутренние документы оператора (приказ, перечень ПДн, модель угроз) — формальности оператора по 152-ФЗ.
